CodeWiki：让 LLM 自动读懂你的代码仓库

发表于 2026-06-10 分类于 工具 阅读次数： 本文字数： 2.1k 阅读时长 ≈ 8 分钟

前阵子接手了一个新项目，打开一看：几千行代码，七八个模块，几十个类互相调用，没有文档，没有架构说明，连个 README 都只有三行。想理清入口在哪，只能从 main.py 一路跳转跟踪下去，半天过去了才勉强画出个草稿。

这种时候就很希望有个东西能帮我把代码结构自动梳理出来。正好看到 CodeWiki——一个用 LLM 自动给代码仓库生成文档的工具。它不是简单地”让 AI 读代码然后写文档”，而是在背后做了一整套依赖分析、模块聚类、Agent 协作的工程化工作。花了点时间研究了一下它的设计，挺有意思的，写篇文章梳理一下。

它做了什么

简单说：输入一个代码仓库（本地路径或 GitHub URL），CodeWiki 会输出一套结构化的 Markdown 文档——包括仓库概览、每个模块的详细说明、Mermaid 架构图，甚至能打包成一个可直接部署的静态站点。

跑完之后，你的 output/docs/ 目录里会有一套完整的文档，打开 index.html 就能像浏览官方文档一样翻阅。

但真正有意思的不是”能做什么”，而是”怎么做”。它的架构设计有几个让我觉得”妙啊”的地方。

第一层：把代码变成”图谱”

任何文档生成的前提，是先搞清楚”这个项目里到底有什么”。

CodeWiki 的做法不是简单地把文件列表扔给 LLM，而是先跑一轮依赖分析。这一步不依赖 LLM，纯静态分析。

它内置了 9 种语言的 AST 解析器：Python 用自带的 ast 模块，JS/TS/Java/C#/C/C++/PHP/Kotlin 则基于 Tree-sitter。跑完后会输出一个完整的有向图——每个节点是一个函数/类/方法，边是调用关系。

这个图的叶子节点（不被任何其他节点依赖的）就是后续文档生成的”起点”。同时这个图还要做拓扑排序和环检测——万一真有循环依赖，要先打破才能往下走。

这个过程很像编译器的前端，只不过目标不是生成 IR，而是为 LLM 提供结构化的上下文。这一步还有一个精细化设计：优先保留 class/interface/struct 类型的叶子节点，如果这些不够才降级取 function。因为类的文档价值比单个函数高得多。

第二层：LLM 聚类——“这些文件应该归到一个模块里”

有了叶子节点列表之后，该怎么组织文档？如果每个节点写一篇文档，228 个组件就是 228 个文件，读者直接劝退。

CodeWiki 的做法是让 LLM 自己聚类。把所有叶子节点列表和它们的代码摘要发给 LLM，LLM 判断哪些节点属于同一个逻辑模块，输出一个模块树：

比硬编码的分组规则灵活太多了。比如一个项目里 utils/ 目录下的文件，有些是公共工具，有些是某个模块专用的辅助函数——只有 LLM 能理解这些语义差异并做出合理分组。

这里有一个聪明的 Token 预算控制：当叶子节点太多时，只发送组件摘要而非完整源码，聚类模型和文档生成模型也可以分开配置——聚类用便宜的模型，文档生成用强模型。

第三层：Agent 三件套

这是整个系统的”灵魂”部分。生成每个模块文档的不是简单的 prompt-completion，而是一个配备了三个工具的 LLM Agent：

  1. read_code_components — Agent 的”眼睛”。按组件 ID 读取任意源码。Agent 初始只拿到核心组件列表，但随时可以按需查更多代码。

  2. str_replace_editor — Agent 的”手”。对文档目录进行文件操作（创建、编辑、插入）。灵感来自 SWE-agent。有意思的是，这个编辑器不允许 Agent 修改源码目录——只读源码，只写文档。还有个小细节：在读文件时，会自动把视图窗口扩展到完整的函数/类边界，避免 Agent 看到”断头”代码。

  3. generate_sub_module_documentation — Agent 的”分身术”。如果当前模块太复杂，Agent 可以递归创建子 Agent 去分别处理每个子模块。

第三个工具是整个体系的关键。它让文档生成变成了动态规划式的处理：先处理叶子模块，等它们的文档写好了，再把子文档内容作为上下文喂给父模块。这样一来，生成 backend.md 的时候，LLM 已经知道 agent_tools.md 和 dependency_analyzer.md 里写了什么，产生的总览文档质量明显更高。

第四层：实时 Mermaid 校验

让 LLM 写 Mermaid 图表有多痛苦，用过的人都知道。少一个空格、箭头符号写错、节点名里有特殊字符——任何一个微小错误都导致渲染失败。

CodeWiki 在 str_replace_editor 的每次文件写入操作后，自动提取 Markdown 中的所有 Mermaid 代码块进行解析校验。用了双保险策略：首选 PythonMonkey + SpiderMonkey JS 引擎（和 Mermaid 官方渲染引擎相同），降级时用纯 Python 的 mermaid-py。

解析失败就报具体行号，Agent 根据错误信息修正重试。这个小设计大幅提升了文档中架构图的可靠性。

第五层：双后端——有 Key 和没 Key 都能用

LLM 调用这块做了个漂亮的抽象。LLMBackend 是一个接口，有两个实现：

  • PydanticAIBackend：API Key 模式，基于 pydantic-ai，支持 OpenAI / Azure / Bedrock / Anthropic，带模型回退链
  • CawBackend：订阅模式，封装 claude / codex 的 CLI，利用用户已有的 OAuth 订阅，不需要 API Key

第二个实现特别实用——如果你已经有 Claude Code 或 Codex 的订阅，直接用就好了，不用额外搞 Key。

CawBackend 还有几个工程细节值得一提：因为 caw 的调用是同步阻塞的，需要用 asyncio.to_thread 推到工作线程；Mermaid 校验又需要回到主事件循环，通过全局注册的方式解决。Codex 的上游库还缺了 tool_timeout_sec 配置，代码里直接做了临时补丁。

增量更新：只生成变了的部分

这个功能的设计很实用。通过 --update 标志：

  1. 读上次生成的 metadata.json，拿到当时的 commit_id
  2. git diff 对比当前 HEAD，找出变更了哪些文件
  3. 在 module_tree.json 中定位受影响的模块
  4. 删除对应的 .md 缓存
  5. 只重新生成这部分文档

对于活跃开发的仓库，避免了每次改几个文件就要全量跑一遍。

最后

CodeWiki 让我觉得有意思的地方，不是它”用 AI 生成文档”这个结果——现在谁都会调 API 做这事。而是在这个目标之下，它用了一套完整的工程手段来保证质量：静态分析做基础、LLM 做语义理解、Agent 工具链做执行、拓扑排序做编排、实时校验做反馈。每一层解决一个具体问题。

从某种意义上说，这也是 LLM 应用的一个缩影：prompt 只是冰山一角，真正让产品可用的，是冰山水面之下的工程基础设施。
